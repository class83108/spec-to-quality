# goals-discovery 評估清單

> 用途：評估 `goals-discovery` skill 是否真的能把混亂對話收斂成可用的 `goals.md`，而不是只看起來問得很完整。

---

## 評估原則

不要先評估：

- 問題數量多不多
- 文件格式整不整齊
- 看起來像不像成熟流程

先評估：

- 能不能把使用者的散亂描述轉成清楚 goal
- 能不能守住正確抽象層
- 能不能自然接到後續設計流程
- 成本會不會高到讓人不想用

---

## 一、對話品質

### 1. 問題是否夠具體

檢查：

- 問題是否能引出具體情境，而不是讓使用者只回答空話
- 問題是否避免「請描述你的系統」這種過大提示
- 問題是否能引出 actor、workaround、failure、scope boundary

好的訊號：

- 使用者開始描述真實工作情境
- 使用者開始說出今天怎麼做、哪裡痛、什麼算有用

不好的訊號：

- 使用者一直停留在「提升效率」「管理資料」「做平台」

### 2. 追問是否有層次

檢查：

- agent 是否先問主幹，再針對缺口補問
- 每輪是否只追 1-3 個最有價值的問題
- 是否有摘要後再繼續，而不是一路連問

好的訊號：

- 對話有收斂感
- 使用者感覺是在一起釐清，不是在被盤問

不好的訊號：

- 問題重複
- 每題都很大
- 問了很多但資訊沒有變清楚

### 3. 是否能把人拉回正確抽象層

檢查：

- 當使用者跳到 tech stack、DB、framework 時，agent 是否能抽出背後意圖
- 當使用者丟 feature list 時，agent 是否能轉回 user problem / required outcome
- 當使用者講太模糊時，agent 是否能逼近可驗證描述

好的訊號：

- implementation detail 被移到 constraints 或 open questions
- feature list 被轉成 candidate goals

不好的訊號：

- goals 區開始出現工具名、框架名、資料庫名

---

## 二、輸出品質

### 4. goals.md 是否真的可用

檢查：

- 讀起來像不像人真的會拿去做後續設計
- 每個 goal 是否描述穩定能力，而不是短期做法
- goals 之間是否彼此清楚、不重疊、不互相衝突

好的訊號：

- 看完後大致知道系統第一版要做什麼
- 看完後也知道哪些不做

不好的訊號：

- 看完還是不知道系統的核心價值路徑
- goals 大量重複或只是換句話說

### 5. non-goals 是否真的有約束力

檢查：

- 每個 non-goal 是否真的排除了某種設計方向
- non-goals 是否有助於防 scope creep

好的訊號：

- 看得出哪些需求就算有人提，這版也不接

不好的訊號：

- non-goals 都是「不做 mobile app」這種幾乎沒有限縮效果的句子

### 6. NFR 是否來自真實壓力

檢查：

- NFR 是否從使用情境、等待感、失敗代價、成本壓力長出來
- 是否有 metric、target、measurement

好的訊號：

- NFR 跟真實使用場景有關

不好的訊號：

- 為了格式而寫一堆 generic NFR
- 寫了 fast / scalable / reliable 但沒有量化

### 7. Open Questions 是否誠實

檢查：

- 有沒有真的把不確定性留下來
- 有沒有把會影響 goals 的未知條件攤開

好的訊號：

- 文件承認未知，但未知是有邊界的

不好的訊號：

- 為了看起來完整，把 Open Questions 寫成空的

---

## 三、抽象控制能力

### 8. 對壞輸入的抵抗力

用這幾種輸入測 skill：

- 很模糊：`我想做一個 AI 協作平台`
- 太技術：`我想用 Postgres、Redis、queue`
- feature list：`要登入、通知、搜尋、權限、報表`
- 太細流程：`按下按鈕後先 call A 再 call B`

檢查：

- 是否能抽出背後問題
- 是否能把細節重新歸位
- 是否能從功能清單中抽出穩定能力

好的訊號：

- 最後仍然能產出像樣的 goals draft

不好的訊號：

- 對話被使用者輸入帶著跑

### 9. 是否能穩定產生 candidate goals

檢查：

- agent 是否主動提出 goal-shaped draft
- draft 是否容易讓使用者 confirm / reject / split

好的訊號：

- 使用者是在修 draft，不是從零寫文件

不好的訊號：

- agent 一直問，但從不整理成候選內容

---

## 四、流程銜接能力

### 10. 是否能自然接到 design-driver-discovery

檢查：

- goals 是否足以支撐下一步找 design drivers
- 是否能看出哪些 flow 可能重要
- 是否能看出哪些 failure impact 值得深入

好的訊號：

- 做完 goals 後，下一步很自然會去問「哪些 flow 與壓力會主導設計」

不好的訊號：

- goals 太抽象，無法接到 flow / pressure

### 11. 是否有助於後續 system-map

檢查：

- goals 是否讓人知道哪些能力需要架構支撐
- 哪些是穩定能力，哪些只是暫時參數

好的訊號：

- 後續可以根據 goals 討論 responsibilities 與 boundaries

不好的訊號：

- goals 太像產品願景文案，無法進設計

---

## 五、成本與摩擦

### 12. 問得細，但不能問得重

檢查：

- 一次 discovery 是否過長
- 是否有太多低價值問題
- 是否為了完整性而犧牲收斂速度

好的訊號：

- 20-40 分鐘內能產出一版有用 draft

不好的訊號：

- 使用者回答到後面只剩疲勞
- 為了走流程而問流程

### 13. 使用者是否感覺在共同建模

檢查：

- 使用者是否感覺 agent 在幫忙整理，而不是在做問卷
- 是否有足夠的摘要、改寫、確認

好的訊號：

- 使用者覺得「你幫我講出了我想表達的東西」

不好的訊號：

- 使用者覺得只是被持續提問

---

## 建議測試方式

至少用 3 種案例試跑：

1. 很模糊的新點子
2. 有既有人工流程要數位化的系統
3. 使用者已帶著很多技術預設的案子

每個案例都評 5 件事：

1. agent 有沒有抓到真正問題
2. goals 有沒有掉進 implementation detail
3. non-goals 有沒有成形
4. open questions 有沒有被挖出來
5. 產物能不能自然接到 `design-driver-discovery`

---

## 快速打分版

每項 1-5 分：

- 對話具體度
- 追問收斂性
- 抽象控制能力
- goal draft 品質
- non-goal 約束力
- NFR 真實性
- open question 誠實度
- 對後續流程的可銜接性
- 整體成本與摩擦

建議判讀：

- `40-45`：可穩定使用
- `30-39`：可用，但有明顯改進空間
- `20-29`：流程表面完整，但實際效果不穩
- `<20`：需要重構 skill，不只是微調 wording

---

## 最後的判準

不要問：

- 這份 skill 看起來專不專業
- 條目列得多不多

要問：

- 它能不能穩定把混亂對話變成可用的 `goals.md`
- 它能不能讓人和 AI 在正確抽象層上合作
- 它能不能支撐後面的 `design-driver-discovery` 與 `SYSTEM_MAP`
