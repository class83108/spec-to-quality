# Systematic Debugging（系統化除錯）

你遇到了一個 bug 或非預期行為。在修改任何 code 之前，你必須先完成診斷流程。

## 核心原則

**絕對禁止猜測性修改。** 如果你不能解釋「為什麼這個改動會修好問題」，你就不該改。

## 測試命令

參照專案 CLAUDE.md 的 Commands 區段取得正確的測試命令。執行測試時使用 `-v --tb=long` 取得完整的 traceback。

## 流程

### Step 1: 定義問題

清楚描述：
- **預期行為**：應該發生什麼？
- **實際行為**：實際發生了什麼？
- **重現步驟**：怎麼觸發這個問題？

### Step 2: 收集證據

根據問題類型收集不同的證據：

**測試失敗：**
- 使用專案的測試命令跑失敗的測試（加 `--tb=long`）
- 讀完整的 traceback，找到實際的錯誤點
- 確認是 test code 的問題還是 production code 的問題

**Runtime 錯誤：**
- 讀錯誤日誌 / traceback
- 檢查相關的 input 資料
- 確認環境狀態（DB、外部服務、設定）

**行為不符預期：**
- 加入 print/logging 確認實際的執行路徑
- 檢查 input → 中間狀態 → output 的每個階段

### Step 3: 縮小範圍

- **找到最小重現**：能用最少的 code/設定重現問題嗎？
- **二分法**：問題是在哪個步驟開始出現的？
- **排除法**：哪些部分確定沒問題？

### Step 4: 建立假說

基於證據，提出 1-3 個可能的原因，每個假說必須：
- 能解釋觀察到的所有症狀
- 是可驗證的（你能設計一個測試來確認或排除它）

將假說呈現給使用者：

> **假說 1**：[描述] — 驗證方式：[如何確認]
> **假說 2**：[描述] — 驗證方式：[如何確認]

### Step 5: 驗證假說

從最可能的假說開始驗證：
1. 執行驗證步驟
2. 如果確認 → 進入修復
3. 如果排除 → 嘗試下一個假說
4. 如果全部排除 → 回到 Step 2 收集更多證據

### Step 6: 修復

1. **最小修改**：只改需要改的，不要順便重構
2. **跑測試**：確認修復有效且沒有 regression
3. **解釋根因**：告訴使用者「為什麼會壞」，不只是「怎麼修」

## Examples

### Example 1: 測試失敗但錯誤訊息不明顯

使用者說：「test_user_registration 失敗了」

1. 定義問題：跑測試看完整 traceback
2. 收集證據：發現 `AssertionError: expected 201 got 400`
3. 縮小範圍：讀 400 的 response body，發現 `{"error": "email already exists"}`
4. 假說：測試之間沒有隔離，前一個測試建立的 user 沒有清理
5. 驗證：單獨跑這個測試 → PASS，全部跑 → FAIL → 確認是隔離問題
6. 修復：加 fixture 確保每個測試有乾淨的 DB 狀態

### Example 2: 反覆嘗試修復超過 2 次

嘗試了兩次修復都沒用。

正確行為：停下來，告訴使用者「我已經嘗試了兩種修法都沒解決，讓我重新從 Step 2 收集更多證據」，而不是繼續猜。

## 反模式（禁止）

- 看到錯誤就直接改 code，沒有先理解原因
- 同一個 bug 用不同的修法反覆嘗試（超過 2 次就停下來重新診斷）
- 修了一個地方壞了另一個地方，進入「打地鼠」模式
- 加一堆 try/except 把錯誤吞掉當作「修好了」

## 卡住時

如果驗證了所有假說都不對，或是問題超出理解範圍：
1. 整理目前收集到的所有證據
2. 列出已排除的假說
3. 明確告訴使用者「我卡住了，目前的理解是 XXX，建議的下一步是 XXX」
